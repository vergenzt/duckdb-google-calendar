# FRD: Google Calendar read/write DuckDB extension (ATTACH catalog)

## Summary
Build the initial implementation of a DuckDB extension that exposes Google Calendar
as an attachable catalog: `ATTACH ... (TYPE google_calendar)` surfaces each calendar
the credential can see as its own table, and users run native `SELECT` / `INSERT` /
`UPDATE` / `DELETE` (and `MERGE` via the planner) against an `events` table. It draws
on the `reference/duckdb_gsheets` extension for its auth, HTTP, and JSON-client
patterns, but goes materially further than gsheets by implementing a real DuckDB
`StorageExtension`/catalog so that genuine SQL DML — not just bulk `COPY TO` — drives
two-way sync with the live Google Calendar API.

## Problem & Intent
The developer is a "two-way sync / automation author": someone who wants to script
calendar mutations from SQL — bulk-creating, updating, and reconciling events via
`INSERT`/`UPDATE`/`MERGE` against a live calendar — where read and write are equally
central. The existing `duckdb_gsheets` reference only supports bulk `COPY ... TO`
(append / clear-then-append) and has no per-row DML, which does not satisfy the
two-way-sync intent. The goal is a SQL-native surface where a calendar behaves like a
mutable table.

## Goals
- Expose Google Calendar as an attachable catalog where each accessible calendar is a table.
- Support native SQL `SELECT`, `INSERT`, `UPDATE`, `DELETE` against an `events` table,
  with `MERGE` working through DuckDB's planner.
- Round-trip full event fidelity by exposing core scalar columns plus nested fields as JSON.
- Reuse the proven `duckdb_gsheets` auth (Secret Manager), HTTP (cpp-httplib+OpenSSL),
  and JSON (nlohmann::json) stack.
- Ship deterministic CI tests (mock HTTP) plus credential-gated live integration tests.

## Non-Goals
- The bulk `COPY ... TO` write path that gsheets uses (superseded by real DML).
- Comprehensive natively-typed modeling of nested fields (attendees/recurrence as
  `LIST<STRUCT>`); nested fields are JSON passthrough in v1.
- Transactional atomicity / rollback across a multi-row statement.
- Client-supplied `iCalUID` idempotent-import workflow (server `id` is the identity in v1).
- Default/implicit time windowing on unbounded `SELECT` (explicitly rejected — see below).

## Functional Requirements
1. The extension SHALL register an `ATTACH ... (TYPE google_calendar)` storage extension
   that enumerates the calendars visible to the credential and exposes each as a table.
2. Each calendar table SHALL expose an `events` schema with core scalar columns
   (`id`, `summary`, `description`, `location`, `start`, `end`, `all_day`, `status`,
   `created`, `updated`, `html_link`) plus nested fields (`attendees`, `recurrence`,
   `reminders`, `conference_data`) as JSON/VARCHAR passthrough.
3. `start` and `end` SHALL be `TIMESTAMP WITH TIME ZONE`; an `all_day` BOOLEAN SHALL
   distinguish date-only events (stored at local midnight).
4. `SELECT` SHALL push `start`/`end` time predicates down to the API's `timeMin`/`timeMax`
   and transparently paginate across all matching pages.
5. `SELECT` SHALL require an explicit time bound; a query with no time predicate SHALL
   raise an error rather than implicitly filtering or paginating the entire history.
6. `INSERT` SHALL create events via the Calendar API (`events.insert`); the server-assigned
   `id` is generated by Google (null/ignored on insert).
7. `UPDATE` SHALL modify events via the API (`events.patch`/`events.update`), matching rows on `id`.
8. `DELETE` SHALL remove events via `events.delete`, matching rows on `id`.
9. `MERGE INTO ... ON ...` SHALL work through DuckDB's planner over the supported
   INSERT/UPDATE/DELETE operators (no bespoke upsert key).
10. Authentication SHALL use a DuckDB Secret of type `google_calendar` supporting
    interactive `oauth`, raw `access_token`, and service-account `key_file` providers.

## Non-Functional Requirements
- **Performance**: Reads paginate fully within the requested time window; no implicit cap.
  Each event mutation is an individual API call (row-by-row), accepted for v1.
- **Security**: Credentials flow exclusively through DuckDB Secret Manager (mirroring
  `duckdb_gsheets`); service-account path does RS256 JWT signing via OpenSSL. No tokens
  in query text.
- **UX / Accessibility**: SQL-native ergonomics — a calendar behaves like a mutable table.
  Unbounded `SELECT` fails loudly with a clear "explicit time bound required" error.
- **Reliability**: Multi-row DML is fail-fast — abort on the first hard API error and
  surface it; already-applied rows remain applied (no rollback). Transient `429`/`5xx`
  responses SHALL be transparently retried with exponential backoff before giving up.

## Constraints & Assumptions
- Builds on the DuckDB C++ extension template currently scaffolded as `waddle`
  (`src/waddle_extension.cpp:29-63`), which must be renamed to the calendar extension.
- HTTP via vendored cpp-httplib + OpenSSL behind an `IHttpClient` interface; JSON via
  nlohmann::json — both modeled on the `reference/duckdb_gsheets` `sheets/` subtree.
- `vcpkg.json` currently declares only `openssl`; nlohmann/json (and any non-vendored
  HTTP deps) must be added.
- Assumes the Google Calendar API supports the catalog enumeration (`calendarList`) and
  event CRUD (`events.list/insert/patch/delete`) endpoints needed; research/design to verify.
- Assumes DuckDB's `StorageExtension`/catalog API is sufficient for dynamic per-calendar
  table enumeration; this is NOT demonstrated in the gsheets reference and must be built from scratch.

## Acceptance Criteria
- [ ] After `ATTACH '<account>' AS cal (TYPE google_calendar)`, `SHOW TABLES` (or catalog
      introspection) lists one table per visible calendar.
- [ ] `SELECT * FROM cal."<calendar>" WHERE start >= TIMESTAMPTZ '...' AND start < TIMESTAMPTZ '...'`
      returns events with core columns typed and nested fields as JSON.
- [ ] `SELECT` from a calendar table with no time bound raises a clear error.
- [ ] `INSERT INTO cal."<calendar>" (...)` creates an event observable on the live calendar
      (verified in the credential-gated live test) and via the mock in CI.
- [ ] `UPDATE ... SET ... WHERE id = '...'` and `DELETE ... WHERE id = '...'` modify/remove
      the targeted event.
- [ ] `MERGE INTO cal."<calendar>" ... ON ...` resolves to insert/update/delete correctly.
- [ ] A transient `429`/`5xx` from the mock HTTP client is retried with backoff; a hard
      error aborts the statement and surfaces the message.
- [ ] `make test` runs the sqllogictests against the mock HTTP client and passes in CI
      without credentials; live integration tests run only when a `google_calendar` secret/env is present.

## Recommended Approach
Implement a DuckDB `StorageExtension` registered for `TYPE google_calendar` that, on
`ATTACH`, calls the Calendar API `calendarList` to enumerate calendars and builds a
catalog with one `events` table per calendar; back the table with custom physical
scan + insert/update/delete operators that translate to `events.list/insert/patch/delete`
calls over a layered transport/resource/auth client subtree ported from
`reference/duckdb_gsheets/src/sheets/`, using a `google_calendar` Secret type for credentials.

**Chosen**: Two-way sync / automation author — scripting calendar mutations from SQL where read and write are equally central.
**Rationale**: Frames write support as core, not a bonus — directly drives the catalog/DML decision below.

### Auth via DuckDB Secret Manager
**Question**: Pre-resolved from codebase evidence — confirmed in Step 4.
**Recommended**: New `google_calendar` secret type mirroring gsheets' oauth / access_token / key_file providers.
**Chosen**: Confirmed.
**Rationale**: evidence: reference/duckdb_gsheets/src/gsheets_auth.cpp:110-148 + confirmed.

### Reuse gsheets HTTP/JSON stack
**Question**: Pre-resolved from codebase evidence — confirmed in Step 4.
**Recommended**: Vendored cpp-httplib + OpenSSL behind `IHttpClient`, nlohmann::json, layered transport/resource/auth client subtree.
**Chosen**: Confirmed.
**Rationale**: evidence: reference/duckdb_gsheets/src/sheets/transport/httplib_client.cpp + confirmed.

### Rename scaffold off 'waddle'
**Question**: Pre-resolved from codebase evidence — confirmed in Step 4.
**Recommended**: Replace the `waddle` scalar-function template (target name, class, entry macro, test) with the calendar extension.
**Chosen**: Confirmed.
**Rationale**: evidence: src/waddle_extension.cpp:29-63 + confirmed.

### Write mechanism — StorageExtension catalog
**Question**: How should writes work? Stated goal is INSERT/UPDATE/MERGE, but gsheets only does bulk COPY TO.
**Recommended**: ATTACH + StorageExtension catalog with native INSERT/UPDATE/DELETE (+MERGE via planner).
**Chosen**: ATTACH + StorageExtension catalog.
**Rationale**: Only path delivering the literal SQL DML semantics requested; accepts highest implementation cost (no reference to copy — must build catalog/transaction/table-entry + physical mutation operators).

### ATTACH model — table per calendar
**Question**: When a user runs ATTACH, what does the resulting schema look like?
**Recommended**: One calendar → one `events` table.
**Chosen**: Account → one table per calendar.
**Rationale**: Enables multi-calendar joins/automation from a single handle; accepts a `calendarList` call at attach time, dynamic catalog enumeration, and name-collision handling.

### v1 operation scope
**Question**: Which operations must work in this initial implementation?
**Recommended**: SELECT foundational; others per intent.
**Chosen**: SELECT, INSERT, UPDATE, DELETE all in v1 (MERGE rides on these via the planner).
**Rationale**: Full two-way DML is the core intent; nothing deferred from the CRUD surface.

### Event identity — server id, MERGE via SQL
**Question**: How should rows be identified for UPDATE/DELETE (and MERGE)? Server id vs client iCalUID.
**Recommended**: Server `id` is the key; MERGE via SQL.
**Chosen**: Server `id` is the key; MERGE via standard SQL `ON`.
**Rationale**: Aligns with the API and keeps v1 simple; accepts no client-driven idempotent insert (iCalUID import out of scope).

### Event schema — core columns + nested JSON
**Question**: How rich should the events table schema be in v1?
**Recommended**: Core scalar columns only.
**Chosen**: Core scalar columns plus attendees/recurrence/reminders/conference_data as JSON passthrough.
**Rationale**: Full-fidelity round-tripping without modeling nested types; accepts weaker type safety on nested fields.

### Datetime model — TIMESTAMPTZ + all_day
**Question**: How should event start/end be represented as columns?
**Recommended**: TIMESTAMPTZ + all_day flag.
**Chosen**: `start`/`end` as TIMESTAMP WITH TIME ZONE; `all_day` BOOLEAN for date-only events.
**Rationale**: Native temporal querying/filtering; accepts loss of the event's original IANA timeZone string on read.

### Read strategy — require explicit time bound
**Question**: How should SELECT handle volume/filtering against huge history? (developer redirected to a custom answer)
**Recommended**: Pushdown + paginate, no implicit cap.
**Chosen**: Push time predicates to timeMin/timeMax + paginate, BUT require an explicit time bound — error if absent (no implicit filter, no accidental full-history scan).
**Rationale**: Avoids both silent bounding and runaway unbounded scans; makes read cost explicit and intentional.

### Write failure handling — fail fast + retry/backoff
**Question**: How should partial failures and rate limits behave for multi-row DML?
**Recommended**: Fail fast, no rollback.
**Chosen**: Fail fast on hard errors (no rollback) + transparent retry with exponential backoff on transient 429/5xx.
**Rationale**: Clear failure signal plus resilience against rate limits; accepts no atomicity (partial writes persist on hard failure).

### Testing approach
**Question**: What testing approach defines 'done' for v1?
**Recommended**: Mock HTTP client + sqllogictests.
**Chosen**: Mock HTTP client + sqllogictests AND credential-gated live integration tests.
**Rationale**: Deterministic CI without credentials, plus real API-contract validation when a secret is present; C++ serialization unit tests not required in v1.

## Open Questions
- Default time-window / cap semantics are intentionally avoided in favor of "require
  explicit bound" — but the exact predicate forms recognized for pushdown (e.g. which
  operators on `start`/`end`, BETWEEN, half-open ranges) are left for design to specify.
- Whether nested-field JSON columns are writable round-trip in v1 or read-only first
  (attendees/recurrence write semantics) — clarify during design.
- Exact retry/backoff parameters (max attempts, base delay, jitter) — left to design.

## Suggested Follow-ups
- gsheets exposes a `COPY ... TO (FORMAT gsheet)` bulk path (`reference/duckdb_gsheets/src/gsheets_copy.cpp:22-27`); a bulk-load convenience for calendars could be a later addition but is explicitly out of v1 scope.
- Client-supplied `iCalUID` idempotent import (`events.import`) for dedupe-on-insert workflows — deferred; revisit if idempotent sync becomes a need.
- Natively-typed nested fields (attendees as `LIST<STRUCT>`, recurrence as `LIST<VARCHAR>`) — ergonomic upgrade beyond v1's JSON passthrough.
